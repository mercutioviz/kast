# `baseline-v3.0-phase-a/` — v3 Phase A render baseline

Snapshot of a complete report render produced at the end of Phase A.
Used as the gold standard for ongoing regression testing while v3
work continues into Phase B and beyond.

## Note on plugin naming

This baseline was captured at Phase A (2026-05-01), **before** the Phase C7 rename of `ai_chatbot_detection` → `ai_surface_detection`. The file `ai_chatbot_detection_processed.json` and its references in `kast_info.json` reflect the old name intentionally — this is historical data, not a bug. The regression test in `test_baseline_render.py` is aware of this and does not assert on the `ai_chatbot_detection` plugin name specifically.

## What this directory contains

A complete kast scan output dir as it would land on disk after
`kast scan -t example.com -o <dir>` (or `--report-only`):

- **Plugin output JSONs** — `<plugin>.json` (raw), `<plugin>_processed.json`
  (post-processed). These were copied from `docs/baseline-v2.14/sample-scan-1/`,
  then regenerated by Phase-A code via `kast --report-only`. Per-plugin
  post-processing has evolved since v2.14.5 was captured; the most
  visible change is `testssl_processed.json` going from 0 issues to 21
  (the testssl plugin's post-processing was upgraded to extract cipher
  and vulnerability findings between the v2.14.5 baseline capture and
  Phase A — a pre-Phase-A change).
- **`kast_report.html`** — rendered by `kast.report.render_html` at end
  of Phase A. The reference for "what an HTML report should look like."
- **`kast_report.pdf`** — rendered by `kast.report.render_pdf`. Reference
  for PDF output.
- **`kast_style.css`** — copied alongside the HTML report by
  `render_html` (so the directory is self-contained).
- **`missing_issue_ids.json`** — present only when the baseline plugins
  emit issue IDs not in `kast/data/issue_registry.json`. After the
  initial baseline capture (commit `4e4dcab`), `BEAST`,
  `BEAST_CBC_TLS1`, and `fallback_SCSV` were promoted to the registry,
  so a clean render against this baseline now produces no missing-
  issues file. If a future plugin emits a new unknown ID, the file
  will reappear here at the next baseline refresh.

## How the regression test uses this directory

`kast/tests/test_baseline_render.py` copies this directory to a temp
location, runs `kast.report.generate_html_report` against the loaded
`*_processed.json` files, and asserts on structural invariants:

- All 10 plugin sections present (`tool-<name>` IDs match)
- Severity counts dict has all 5 canonical Severity enum keys
- No `<wbr>` tags (A9 invariant)
- Every `href="#tool-X"` resolves to an `id="tool-X"` (A8 macro invariant)
- Issue count matches what `collect_report_data` produces against the
  baseline JSONs

The test does NOT byte-diff against `kast_report.html` here — that
file's bytes contain timestamps and could drift on cosmetic changes.
The structural assertions catch the regressions we actually care about.

## What this baseline does NOT cover

- **End-to-end CLI invocation** — the test calls the Python report
  pipeline directly. A future test should invoke `python -m kast.main
  --report-only ...` via subprocess to also exercise the CLI dispatch,
  argparse, registry construction, and orchestrator paths.
- **kast-web integration** — verifying that kast-web reads these
  artifacts correctly is a separate, manual test (see
  `docs/web-integration.md`).
- **Active-mode ZAP scans** — no ZAP data is present.

## Refreshing the baseline

If a deliberate change makes the current baseline outdated:

```bash
rm -rf /tmp/v3-validation
cp -r docs/baseline-v3.0-phase-a /tmp/v3-validation
rm -f /tmp/v3-validation/kast_report.* /tmp/v3-validation/kast_style.css /tmp/v3-validation/missing_issue_ids.json
.venv/bin/python -m kast.main --report-only /tmp/v3-validation/ --format both
cp -r /tmp/v3-validation/* docs/baseline-v3.0-phase-a/
```

Then commit with a message explaining what changed and why.
