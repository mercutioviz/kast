# kast v3.1+ backlog

Capability candidates deferred from the v3.0 ship. The Tier numbers refer to the prioritization framework in `v3-planning/02-ideation.md`. The IDs (A2, A6, D1, etc.) match the ideation menu.

This list is forward-looking, not committed. Prioritization happens per release.

## v3.1 candidates

- **A2 — Pre-meeting SA briefing** (Tier 2). Generate talking points from a completed scan, tuned to the meeting audience. Compelling but a new artifact type; depends on A1 (the LLM exec summary) being stable in the field.
- **A3 — Tailored remediation by audience** (Tier 2). Per-audience variants (exec / dev / ops) of the same finding. Compounds well with A1 but roughly doubles AI cost per scan.
- **F3 — Shareable scan-result URL** (Tier 2). Generate a signed URL that exposes a scan's report to a prospect or partner without a kast-web login. Useful for follow-up after demos.
- **F4 — Per-partner theming** (Tier 2). Brand the report for partner sales engineers — logo, color palette, footer copy. Larger surface than it sounds because PDF rendering is involved.
- **E5 — Findings diff between scans** (Tier 2). "What changed since last scan." Direct sales hook for "before WAF / after WAF" comparison demos.
- **A6 — "What changed since last scan" AI narrative** (Tier 3). Natural-language summary built on E5. Depends on E5 landing first.
- **D1 — Expose kast as MCP server** (Tier 3). Strategic but premature for v3.0; revisit as the agent ecosystem matures.
- **D2 — "Ask the report" agent** (Tier 3). Chat UI or MCP over a completed scan. Depends on D1 if MCP is the chosen surface.
- **C1 — Adaptive scan plan** (Tier 3). Let the AI pick plugins based on a recon pass. Real risk of bad LLM decisions choosing the wrong scans; defer until A1 quality is well-understood.

## v3.2+ / unlikely

- **E7 — Multi-target scan** (Tier 4). Scan a list of targets in one invocation. Edges toward continuous monitoring; product-shape question.
- **F5 — Industry benchmarking** ("your prospect vs. their sector") (Tier 4). Requires a substantial corpus of comparable scans; premature.
- **Z1–Z4** — Continuous monitoring, exploit demo, white-label SaaS, internal benchmarking. Out of scope unless the product shape changes.

## Plugin candidates (not in original ideation; surfaced post-ship)

- **CDN/edge fingerprint plugin** (Akamai / Cloudflare / Fastly / others). Direct sales hook for WAF positioning.
- **API surface detection** — OpenAPI/Swagger/GraphQL endpoints exposed. Pairs thematically with `ai_surface_detection`; growing 2026 attack surface.
- **Auth/SSO discovery** — OIDC/SAML endpoint detection beyond what FTAP currently covers.
- **Bot/scraper exposure signals** — rate-limit headers, captcha presence, anti-automation hints. Sales hook for bot-mitigation modules.
- **Third-party script inventory** — supply-chain surface beyond the current `script_detection` plugin.

## How to use this file

When planning a v3.1 (or later) release:

1. Pick a small set of items from this list — Tier 2 candidates are the natural starting points.
2. Re-validate the gating questions for each: AI cost impact, content ownership (some require Barracuda product marketing), kast vs kast-web split.
3. Move the picked items into a new `docs/v3-planning/` doc (or directly into CLAUDE.md as Open threads if scope is small).
4. Update this file to remove the picked items.

When new ideas surface mid-release, add them here so they're tracked but not actioned mid-flight.
